# VPC
- AWS上に自分専用のプライベートネットワークを作る仕組みをVPCという
- VPCはリージョン単位で作成する
- VPCを作成するときにCIDRブロック（IPアドレスの範囲）を指定する

### サブネット
- VPC内はサブネットに分割して使う
- サブネットはアベイラビリティゾーン単位で作成する
- インターネットと通信できるサブネットをパブリックサブネットという
- インターネットと通信できないサブネットをプライベートサブネットという
- パブリックサブネットにするにはインターネットゲートウェイをVPCにアタッチする
- インターネットゲートウェイはVPCとインターネットをつなぐ出入り口のこと
- パブリックサブネットにするにはルートテーブルに0.0.0.0/0のルートを追加する
- ルートテーブルはサブネット内のトラフィックをどこに向けるかを定義するもの
- パブリックサブネット内のEC2は自身のPublicIPで外部通信できる

### NATゲートウェイ
- プライベートサブネット内のEC2がアウトバウンド通信するにはNATゲートウェイを使う
- NATゲートウェイはパブリックサブネットに配置する
- NATゲートウェイはアウトバウンド専用で、外からのインバウンド通信は受け付けない
- NATゲートウェイはAWS管理のマネージドサービス
- NATインスタンスはEC2で自前で建てるNAT（コスト安だが管理が必要）
- NATゲートウェイはAZごとに1つ作成するのが高可用性な構成
- AZをまたいだNATゲートウェイ利用はAZ障害時にプライベートサブネットの通信が止まるリスクあり
- NATゲートウェイはIPv4専用

### セキュリティグループ
- ステートフルは行きの通信を許可すれば戻りの通信は自動で許可される仕組み
- セキュリティグループはEC2などのリソースレベルのファイアウォールでステートフル
- セキュリティグループはデフォルトで全インバウンドを拒否、全アウトバウンドを許可
- セキュリティグループはインスタンス単位で制御する
- セキュリティグループは1つのEC2に複数アタッチできる
- セキュリティグループはルールに「許可」しか設定できず「拒否」は設定できない
- セキュリティグループのインバウンドルールは通常IPアドレスで指定する
- セキュリティグループのインバウンドルールに別のSGのIDをソースに指定できる
- EC2のIPは再起動すると変わってしまう
- セキュリティグループのインバウンドルールにSGID指定すると、IPが変わっても自動で許可される

### ネットワークACL
- ステートレスは行きと戻りの通信をそれぞれ別に許可ルールを設定する必要がある仕組み
- ネットワークACL（NACL）はサブネットレベルのファイアウォールでステートレス
- ネットワークACL（NACL）はデフォルトですべての通信を許可する
- NACLはサブネット単位で制御する
- NACLのルール番号は100刻みで設定するのが慣習
- NACLはルール番号の小さい順に評価され最初にマッチしたルールが適用される

### VPCピアリング
- VPC同士を接続する仕組みをVPCピアリングという
- VPCピアリングは同一リージョン・クロスリージョン・クロスアカウントでも利用できる
- VPCピアリングはA-B-CのときAとCは通信できない
- CIDRが重複するVPC同士はピアリングできない
- ピアリング後は両側のルートテーブルに相手のCIDRへのルートを手動追加する必要がある

### Transit Gateway
- 複数VPCをつなぐときはTransit Gatewayが適している
- Transit GatewayはVPCやオンプレ、VPN接続、Direct Connectをハブで一括管理できる

### VPN接続
- VPN接続は自社オフィスやデータセンター（オンプレ）とAWSをつなぐ仕組み
- VPNとはインターネット回線を使って安全に通信するトンネルのこと
- 仮想プライベートゲートウェイはAWS側に設置するVPN接続の出入り口
- 仮想プライベートゲートウェイはアタッチして利用する
- カスタマーゲートウェイはオンプレ側の実際のルーターや機器のこと
- カスタマーゲートウェイはAWSにその機器のIPアドレスや設定を登録したもの
- 仮想プライベートゲートウェイとカスタマーゲートウェイの間にVPNトンネルを張る
- VPNトンネルを張ることでオンプレとAWS上のVPCやその内部のEC2などが通信できる
- オンプレとVPCを専用線でつなぐにはDirect Connectを使う

### VPC Flow Logs
- VPC Flow LogsとはVPC内を流れた通信記録(誰/どこ/ポート)をログとして残す仕組み
- VPC Flow Logsの保存先はCloudWatch LogsかS3を選べる
- VPC Flow LogsはリアルタイムではなくCloudWatchやS3に数分遅れで届く

### デフォルトVPC
- デフォルトVPCとはAWSアカウントを作ると各リージョンに自動でVPCが1つ用意されている
- 何も設定しなくてもすぐEC2を起動できるのはデフォルトVPCのおかげ
- デフォルトVPCのCIDRは172.31.0.0/16と固定されている

### VPCエンドポイント
- VPC内のEC2がS3にアクセスするにはインターネット(外部通信)経由のためリスクがある
- VPCエンドポイントを使うとAWS内部の専用ルートでS3などにアクセスできる
- VPCエンドポイントとはVPC内のリソースがAWSサービスに接続する仕組み
- VPCエンドポイントはリージョン内のサービスへの接続に使う
- AWSにおける「エンドポイント」は接続口を指している
- VPCエンドポイントにリソースポリシー(エンドポイントポリシー)をアタッチできる
- エンドポイントポリシーはS3の特定バケットへのアクセスのみ許可するといった制御ができる

### ゲートウェイ型エンドポイント
- EC2はVPCの中に置くサーバーだが、S3やDynamoDBはVPC外にあるAWS管理の共有サービス
- 機密データをS3やDynamoDBに保存するとき通信がインターネットに出てしまうとリスク
- S3とDynamoDBはAWSの中でも特に使われる頻度が高いためゲートウェイ型エンドポイントが存在する
- S3とDynamoDBはゲートウェイ型エンドポイントの通信が推奨されている
- VPCにゲートウェイ型エンドポイントを作成すると、ルートテーブルに自動で経路が追加
- ルートテーブルとはサブネット内の通信をどこに向けるかを書いた地図のようなもの
- ゲートウェイ型エンドポイントを使うとインターネットゲートウェイなくてもS3/DynamoDBに接続可
- ゲートウェイ型エンドポイントを使うとNATゲートウェイがなくてもS3/DynamoDBに接続可
- ゲートウェイ型エンドポイントの通信はオンプレから使えない
- ゲートウェイ型エンドポイントの通信は無料
- ゲートウェイ型エンドポイントの通信はセキュリティグループ非対応

### インターフェース型エンドポイント(Private Link)
- S3とDynamoDB以外はインターフェース型の通信のみ
- インターフェース型エンドポイントを作成すると、VPCのサブネット中に自動的にENIが作られる。
- インターフェース型エンドポイント作成時に作られたENIに自動的にプライベートIPが割当てられる
- ENIに割り当てられたプライベートIP宛に通信することでAWSサービスに届く
- インターフェース型エンドポイントの通信は有料
- インターフェース型エンドポイントの通信はセキュリティグループ対応

### エンドポイントサービス
- サービスを別のVPCに安全に公開したいときにPrivateLink + NLB構成を利用する
- VPCピアリングはCIDRが重複するとつなげないため、PrivateLink + NLB構成を利用する
- PrivateLinkは特定のサービスだけを外部に公開できる
- PrivateLinkはNLBとしか組み合わせられない（ALBは不可）
- サービスを提供するVPCにNLBを置き、NLBをもとにエンドポイントサービスを作成する
- エンドポイントサービスは自分のサービスをPrivateLink経由で他VPCに公開する仕組み
- サービス提供VPCは、利用者のAWSアカウントIDに利用許可を出す必要がある
- サービスを利用するVPCにPrivateLinkを作成、作成されたENIのプライベートIP宛に通信できる

### CIDR
- CIDRのプレフィックス長はVPCを作るときにIPアドレスの範囲の広さを/16〜/28の間で指定
- CIDRのプレフィックス長の/16は約65000個のIPアドレスが使える最大サイズ
- CIDRのプレフィックス長の/28は16個のIPアドレスしかない最小サイズ
- VPCのCIDRのプレフィックス長が/16ならサブネットは/17〜/28の間で自由に切れる
- サブネットを作るとAWSが自動で5つのIPアドレスを内部利用のために確保する
- 10.0.0.0/24のサブネットを作った場合、10.0.0.0はサブネット自体のネットワークアドレス
- 10.0.0.0/24のサブネットを作った場合、10.0.0.1はVPCルーター
- 10.0.0.0/24のサブネットを作った場合、10.0.0.2はDNSサーバー
- 10.0.0.0/24のサブネットを作った場合、10.0.0.3は詳細の機能拡張のためにAWSが確保
- 10.0.0.0/24のサブネットを作った場合、10.0.0.255はネットワーク内全体へのブロードキャスト用

### Egress-Only Internet Gateway
- IPv6とはIPv4が枯渇してきたため作られた新しいIPアドレスの形式
- IPv6のアドレスはすべてパブリックIPのため外部から直接アクセスされる可能性がある
- EC2にIPv6を使いつつ外からの通信は受け付けたくない場合にEgress-Only Internet Gatewayを使う
- Egress-Only Internet GatewayはEC2→インターネット通信は許可するが外→EC2通信は遮断する

### Elastic IP
- EC2を停止・再起動するとパブリックIPが変わる
- 固定のパブリックIPが必要なときはElastic IPを使う
- NATゲートウェイを作成するときもElastic IPが必要になる

### Direct Connect
- Direct ConnectはVPNより安定・高速だが開通までに数週間〜数ヶ月かかる
- Direct Connect Gatewayを使うと複数リージョンに接続できる

### Elastic Network Interface(ENI)
- ENIはEC2に取り付けるネットワークカードのようなもの
- ENIを別のEC2に付け替えることでIPを引き継いで障害対応できる
- IPを引き継いで別インスタンスに切り替えることをフェイルオーバーという

### Route 53 Resolver
- Route 53 ResolverはVPCとオンプレ間のDNS名前解決をする
- DNS名前解決とはドメイン名をIPアドレスに変換する仕組みのこと
